# i18n — English-default localization

Argos's README, brand, and model-facing system prompt are English, but the
runtime UI/CLI was originally hardcoded Chinese — so an English-funnel user
could not read the honesty moat (verdict badge, trust dial, ledger), the CLI
help, or the setup wizard. This document describes the minimal i18n layer that
makes the user-facing surface English by default while keeping Chinese
available.

## How it works

- **`argos/i18n.py`** — `t(key, **kwargs)` looks up `key` in the catalog for the
  current language, falls back `en → key`, formats with `str.format`, and never
  crashes on a missing key or a bad format (honest degradation).
  - Language comes from **`ARGOS_LANG`** (default `en`; `zh` supported;
    `zh_CN` / `en-US` / unknown values normalize to `en`/`zh`). Read dynamically
    per call, so switching locale takes effect immediately.
  - `is_error_result(s)` — a locale-independent helper that returns `True` when a
    tool result starts with a known error prefix (`错误:` / `Error:`). Control-flow
    discriminators (`loop.py` ExitPlanMode detection, `dom_probe.py`) use it so
    they keep working when tool errors render in English.

- **`argos/locales/*.py`** — one catalog module per area, each exposing two
  module-level dicts `EN` and `ZH` (`key -> template string`). They are merged at
  first use.
  - Discovery uses the explicit **`CATALOG_MODULES`** list in
    `argos/locales/__init__.py` (PyInstaller-frozen safe — `pkgutil` discovery
    returns nothing inside a frozen bundle), plus a `pkgutil` fallback for dev.
    **Adding a new catalog module requires adding its name to `CATALOG_MODULES`.**
  - Keys are dotted and namespaced by area (`cli.*`, `tui.*`, `widget.*`,
    `verdict.*`, `trust.*`, `ledger.*`, `loop.*`, `sandbox.*`, `daemon.*`,
    `tools.*`, `perm.*`, `config.*`, `common.*`, …).

### The verbatim-ZH contract

Each `ZH` value is the **exact original string, verbatim** (same characters,
punctuation, and glyphs). `tests/conftest.py` defaults `ARGOS_LANG=zh`, so the
~290 existing Chinese-asserting tests keep passing unchanged: under `zh` every
`t()` call renders the original string. New tests cover the English default,
locale switching, and the `EN`/`ZH` key-alignment guard
(`tests/test_i18n.py::test_en_zh_catalogs_have_same_keys`).

## What is localized

**The whole repository is bilingual.** Every string that is displayed, printed,
returned, raised, or read by a human or the model goes through `t()` — the
interactive UI/CLI, the honesty-moat widgets, tool results, and all the deeper
subpackages (`verify`, `perception`, `workflow`, `mcp_native`, `lsp`, `hooks`,
`capability`, `conductor`, `permissions`, `routing`, `memory`, `context`,
`skills*`, `eval`, `learning`, `daemon`, `core`, …). Catalogs are split by area
under `argos/locales/` and registered in `CATALOG_MODULES`.

## What deliberately stays Chinese (data, not messages)

Only strings that are **matching/parsing logic or markup**, never shown as
prose, are left as Chinese literals — translating them would break behavior:

- **matching word-lists / regex patterns** compared against the model's output
  or the user's goal text: `loop.py` lazy-claim markers, `verify/strategy.py` and
  `contracts.py` task-type/domain regexes, `permissions/hard_rules.py`
  financial/payment patterns, `memory/auto.py` scope keywords, `learning/dream`
  and `skills.py` tokenizer ranges
- **control-flow sentinels** that anchor on structured output — these are kept in
  sync with their localized producers, not hardcoded: `context/prune.py` derives
  its bucket markers from the same `loop.exec.*` / `loop.todos.header` i18n keys;
  `routing/config.py` keys off `e.__cause__` (a `json.JSONDecodeError`) instead of
  matching the localized message; `is_error_result` matches the `错误:` / `Error:`
  prefix in either language
- `DEFAULT_CSS` strings and a couple of developer-only constant docstrings

The leak scanner (`scripts/_i18n_leak_scan.py`) reports ~54 such strings across
the repo; each is an intentional skip of one of the above kinds.

### Cross-locale discriminators

When code branches on a string that is now localized, it must **not** match the
Chinese (or English) text directly — that breaks under the other locale. Use one
of: the exception type / `__cause__` (`routing/config.py`), the locale-independent
`is_error_result` helper, or markers derived from the *same* i18n key as the
producer (`context/prune.py`). Both producer and matcher run in one process under
one `ARGOS_LANG`, so key-derived markers always align.

## Maintenance

- Find leaks: `python3 scripts/_i18n_leak_scan.py <file.py ...>` lists Chinese
  string literals that are not docstrings, `t()`/`_t()` args, or `log.*` args —
  i.e. likely-unlocalized user-facing strings. Semantic false positives (CSS
  blocks, matching word-lists) need a human eye.
- When adding a user-facing string: add `EN`/`ZH` entries to the appropriate
  catalog (keep the key sets identical) and call `t("ns.key", ...)`. For an error
  result that must be detected by `is_error_result`, make the `EN` value start
  with `Error:` and the `ZH` value start with `错误:`.
