# Argos — 产品定义(v3,基于实测证据)

> 一句话:**让便宜模型也能在结构化工程任务上可靠协作的多 agent 工作台。**
> 站在 Hermes 之上,不重造轮子;核心是一份工程化的「契约模板」,
> 把傻模型蜂群的产出从「拼不起来」变成「零冲突可直接组装」。

本文档的每个论断都附了实测证据(脚本可复现),不是愿景陈述。

---

## 1. 它解决的真实痛点

- 开发者在 3+ AI 工具间切换(实测:59% 开发者用 3+,20% 用 5+),换工具丢上下文。
- 国内用户大量使用便宜模型(MiniMax / GLM / Kimi / DeepSeek),**质量不稳是公认痛点**——用不起 Opus 4.8。
- Claude Code 的 Dynamic Workflows(目标→拆任务→蜂群→交叉验证)只在 Claude 生态可用,且 70% 能力依赖 Opus 4.8 模型本身。

**Argos 的命题**:不靠换更强的模型,靠一层工程,让便宜模型也能跑出可靠的多 agent 协作结果。

---

## 2. 核心发现(实测,可复现)

用本地 Hermes 接的 **MiniMax-M2.7**(国产便宜模型)做 A/B/C 对照,让 3 个互不通信的
agent 协作设计一个 REST API。脚本:`scripts/swarm-ab-minimax.py`。

| 组 | 做法 | 硬冲突数 |
|---|---|---|
| A | 裸蜂群,各自闷头干 | **8** |
| B | 模型自己自由写契约 | **6** |
| C | 工程化完整契约模板 + 强制对齐自检 | **0** |

**结论**:傻模型不笨,缺的是一份好的「填空题」。给一份覆盖完整的契约 checklist
(ID/命名/时区/枚举vs布尔/并发持久化/封装/长度/接口-模型对齐自检),傻模型产出可达
强模型级的可组装质量。

第二个领域(DB Schema,三张外键关联的表)用同一模型复验:**硬冲突 0**,外键类型/
命名/主键策略全部对齐。两个领域的真实证据,非单点。

---

## 3. 边界(同样实测,诚实标注)

对抗实验(`scripts/swarm-domain2.py`):让蜂群协作「写一篇工具横评文章」(软任务)。

| 组 | 硬不一致数 |
|---|---|
| A 裸蜂群 | 10 |
| C 写作契约模板 | **15(更差)** |

**契约层只能锁住「形式约定」(字段/类型/格式/枚举),锁不住「内容一致性」(谁评了
哪个工具/立场/事实对齐)。** 写作几乎全是内容一致性 → 失败,且契约让模型更啰嗦、
给出更多可矛盾的表面。

**所以 Argos 只做结构化工程任务,不碰开放式内容生成**(后者本就该交单个聊天模型)。
这与「非技术用户要聊天+文件输出不要蜂群」「唯一真用并行 agent 的是 senior 工程师」
两条市场证据吻合。

## 4. 架构:站在 Hermes 之上

```
┌─────────────────────────────────────────────┐
│  Argos(新做的)                                │
│  ① 契约模板引擎  src/engine/contracts.ts        │
│     5 领域 checklist:REST API / DB Schema /     │
│     状态机 / 配置 / 通用,按目标自动分类           │
│  ② 蜂群编排      src/engine/swarm.ts            │
│     目标→契约冻结→并行蜂群→交叉验证               │
│  ③ 冲突诊断可视化 src/components/SwarmPanel.tsx  │
│     不是动画,是「契约冲突高亮 + 可组装判定」       │
├─────────────────────────────────────────────┤
│  Hermes(复用,不重造)  src/lib/llm.ts 走桥      │
│  模型层:OpenRouter/Gemini/GLM/Kimi/MiniMax     │
│  + 391 skills + 记忆 + 自动化 + 多平台投递        │
└─────────────────────────────────────────────┘
```

`src/lib/llm.ts` 的 model-agnostic `ChatFn`:Tauri 下走 `hermes_post → /v1/chat/completions`
(即本地 Hermes 已接的任意模型),浏览器下走 mock。Argos 不自己接模型。

---

## 5. 护城河

由浅到深:

1. **可视化**(浅,可抄):好看的三阶段诊断界面。
2. **蜂群编排**(中,LangGraph 也能做):目标→拆解→并行→验证。
3. **契约模板库**(深,真护城河):那份「如何让傻模型不漏项」的工程知识,是跑实验、
   踩坑、一个领域一个领域积累出来的。**抄不走,因为别人没跑过这些实验。** 覆盖的
   结构化领域越多(API/schema/状态机/配置/迁移/IaC/协议…),护城河越深。

---

## 6. 诚实的风险

- **领域逐个挖**:每个结构化领域的契约 checklist 不同,要一个个验证、积累。这是真实
  工作量,也正是护城河来源。
- **不直接产生收入**:按当前目标(用户优先、付费随缘),这不是现阶段问题;但需清醒。
- **Anthropic 可能下放 DW 到 API**:若 Claude 把 Dynamic Workflows 开放给任意模型,
  会削弱「非 Claude 蜂群」的稀缺性——但「让便宜模型可靠」这个工程价值仍在(DW 仍需强模型)。

---

## 7. 复现实验

```bash
# 前提:本地 Hermes 在 127.0.0.1:8642,API_SERVER 已启用,key 在 ~/.hermes/.argos_api_key
python3 scripts/swarm-ab-minimax.py ab   # REST API:A/B 对照(裸蜂群 vs 自由契约)
python3 scripts/swarm-ab-minimax.py c    # REST API:工程化契约模板(应得 0)
python3 scripts/swarm-domain2.py         # 软领域对抗(应失败,证明边界)
```
